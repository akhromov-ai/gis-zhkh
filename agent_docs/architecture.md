# Архитектура

> Документ описывает архитектуру **продукта/платформы Оператора ЕСД** (а не структуру этого репозитория). Источники: презентация «Старт работ по реализации нового сервиса», документы «План», «Письмо-инициатива». Финансовая часть стратегии (выручка, EBITDA, оценка компании) — в NDA-материалах вне репозитория.

## Обзор

Оператор «Единой Среды Доступа» (ЕСД) — федеральная платформа цифрового контроля доступа в жилые дома и на придомовые территории. Ядро — электронный журнал пропусков, выводимый пользователю как кнопка **«Заказать пропуск»** в приложении **Госуслуги.Дом** и интегрированный с государственной системой **ГИС ЖКХ**. Платформа объединяет:

- электронный учёт заявок и пропусков (журнал);
- программно-аппаратный комплекс (ПАК) для УК — BLE-домофоны, контроллеры, считыватели;
- партнёрскую сеть монтажников (биржа, сертификация, LMS, Service Desk);
- интеграции с поставщиками заявок на доступ (Yandex GO/Маркет, OZON, WildBerries, Купер, Самокат, Delivery, Dostavista и др.);
- открытое API для производителей домофонов («Госуслуги.ДОМ Ready»).

## Контекст

- **Оператор ЕСД** — отдельное юридическое лицо (см. PDF «Юридическая структура Оператора ЕСД»), взаимодействующее с Оператором ГИС ЖКХ и Оператором приложения Госуслуги.Дом по соглашению о совместной работе.
- **Госзадача:** ограничение и упорядочивание доступа в жилой фонд (антитеррор), цифровизация ЖКХ, программа «Умный город».
- **Ключевые внешние системы:** ЕСИА (аутентификация), ГИС ЖКХ (адресные справочники, данные об УК и жителях, места доступа, заявки), Госуслуги.Дом (UI для жителя), партнёрские маркетплейсы и сервисы доставки/такси.
- **Целевая аудитория:** жители многоквартирных домов и закрытых посёлков, управляющие компании, охрана/диспетчеры, сервисные и курьерские службы, производители домофонов, региональные администрации.

## Ключевые компоненты

1. **UI «Заказать пропуск»** — встроенный в Госуслуги.Дом раздел: создание разовых и постоянных пропусков, домовой чат, шаблоны ОСС.
2. **Электронный журнал пропусков (ЕСД-сервис)** — серверная часть Оператора ЕСД: хранение заявок на доступ, статусов, истории проходов; web-кабинет для УК; e-mail/уведомления УК.
3. **API-шлюз ЕСД** — двусторонние интеграции:
   - с **ГИС ЖКХ** (доработка API): адресные справочники, данные об УК, места доступа (шлагбаумы, подъезды, калитки), данные о жителях;
   - с **Госуслуги.Дом** (доработка API + UI и шаблонов документов);
   - с **поставщиками заявок** (Yandex, OZON, WildBerries и др.) — приём заявок на доступ от внешних B2B-сервисов.
4. **ПАК (программно-аппаратный комплекс)** — BLE-модули, контроллеры, считыватели, устанавливаемые на домофоны/шлагбаумы. Поддерживает:
   - автоматическое открытие по приближению (телефон/метка);
   - прямое соединение «смартфон ↔ контроллер» по Bluetooth;
   - обмен данными по криптозащищённому запатентованному протоколу;
   - «сквозной» разовый временный ключ для курьеров; постоянные ключи с привязкой к мастер-ключу.
5. **Облачная СКУД** — для УК без собственной системы доступа (организация постоянного доступа жильцов/сотрудников).
6. **Портал заявок на доступ** — приём заявок от жителей через мобильное приложение/портал УК и от внешних сервисов.
7. **LMS и курсы по устройствам, Service Desk** — обучение и поддержка монтажников.
8. **Управление монтажными бригадами + тесты при монтаже** — сертификация и контроль качества установки.
9. **Биржа монтажников** — маркетплейс сертифицированных подрядчиков и комиссия за установку ПАК.
10. **Шаблоны документов** — типовой договор с УК, шаблон решения ОСС, протокол о намерениях (Term Sheet), соглашение о совместной работе с Оператором информационной системы.

## Потоки данных

### Базовый сценарий «Заказ пропуска жителем»

1. Житель в Госуслуги.Дом нажимает «Заказать пропуск» → формирует пропуск.
2. Запись попадает в **журнал пропусков ЕСД**; жителю/гостю выдаётся идентификатор/ключ.
3. ЕСД отправляет **e-mail УК** с информацией о необходимости обеспечить доступ.
4. УК фиксирует обработку в журнале; информация о проходе возвращается жителю.
5. Накопление неисполненных пропусков и e-mail-запросов на адрес УК **создаёт давление на УК** и приводит её к регулярной работе с журналом, а далее — к ОСС по подключению Журнала пропусков и ПАК (см. § «Стратегия входа на рынок»).

### Стратегия входа на рынок (важное уточнение модели работы)

Сервис может стартовать **до подключения УК** к платформе:
- жители уже могут заказывать пропуски через Госуслуги.Дом, ЕСД автоматически отправляет e-mail УК;
- накопление просроченных пропусков и поток e-mail-запросов на адрес УК — экономически вынуждают УК начать работать с журналом;
- недовольство жителей неисполненными пропусками **создаёт инициативу собственников** на ОСС по подключению Журнала пропусков и ПАК.

Из этого следует архитектурное требование: модули `passes-svc`, `notifications-svc` и `oss-svc` готовятся **раньше** полного onboarding-флоу `uk-svc`, чтобы воронка через жителей могла стартовать без UK-инициативы. ОСС-процесс должен быть максимально автоматизирован — шаблоны решений, методички вовлечения жителей, цифровое голосование через Госуслуги.Дом с ЕСИА-подписью.

### Сегментация УК (двe модели интеграции)

Целевая аудитория «УК» неоднородна. В рамках архитектуры выделяются два сегмента:

| Сегмент | Поведение | Модель интеграции |
|---|---|---|
| **УК с собственной СКУД** | Имеет свою систему доступа и не хочет от неё отказываться | Получает заявки из ЕСД через мобильное приложение / портал УК; ЕСД пишет в журнал, УК закрывает пропуски в своей СКУД |
| **УК без собственной СКУД** | Не имеет систематической СКУД либо хочет уйти от частичных решений | Использует **облачную СКУД ЕСД**: портал заявок + групповое администрирование постоянных ключей жильцов и сотрудников, выдача через мобильное приложение |

Это разделение влияет на onboarding-плейбук (`operations-playbooks.md`), на матрицу ролей (PROPERTY_MANAGER одинаковая, но подмодули доступа разные) и на тарифную модель (см. `NDA/tariff-model.md`).

### Сценарий «Курьер/такси/доставка»

1. Поставщик заявки (Yandex, OZON, WildBerries и т. п.) при оформлении доставки обращается в **API ЕСД** за разовым ключом доступа.
2. ЕСД формирует «сквозной» временный ключ (подъезд, калитка, шлагбаум) и отправляет его в приложение курьера.
3. На месте — открытие через BLE/контроллер; при необходимости — проверка личности охраной через ЕСИА (Госуслуги).
4. Журнал прохода фиксируется в ЕСД; снижается риск конфликтов и хищений.

### Сценарий «УК без собственной СКУД»

- Подключение к **облачной СКУД ЕСД** → выдача постоянных ключей жильцам и сотрудникам, групповое администрирование и обновление ключей через мобильное приложение / портал УК.

### Сценарий «УК с собственной СКУД»

- Заявка на доступ от жителя через Госуслуги.Дом → ЕСД ведёт журнал пропусков и передаёт уведомления УК.
- УК принимает заявки через мобильное приложение / собственный портал УК и закрывает пропуска в своей СКУД.
- Журнал ЕСД остаётся «истиной» для жителя и регуляторных целей (антитеррор, аналитика, audit-trail).

### Сквозные потоки

- **ОСС-поток:** инициация → шаблон голосования → решение собственников → подключение УК к журналу/ПАК. Должен быть максимально автоматизирован (шаблоны решений).
- **Монетизационный поток (структурно):** SaaS-подписка УК + разовая выручка от ПАК + комиссия биржи + B2B-интеграции с поставщиками заявок. Конкретные тарифы и цены — в NDA-материалах.

## Технологии и зависимости

- **Идентификация:** ЕСИА (Госуслуги).
- **Каналы доступа:** Bluetooth Low Energy (BLE), запатентованный криптозащищённый протокол обмена «смартфон ↔ контроллер», облачная СКУД.
- **Концепт UX «нулевое время доступа»:** автоматическое открытие по приближению (для смартфонов и пассивных меток) + прямое BLE-соединение «смартфон ↔ контроллер» без обязательного интернета на стороне устройства в момент прохода + криптозащищённый протокол. Технические детали оффлайн-валидации и кэша ключей в EEPROM — `functional-spec.md` (модули `acl-svc`, `device-gateway`).
- **Интеграции:** API ГИС ЖКХ (доработка), API Госуслуги.Дом (доработка), API партнёров-поставщиков заявок.
- **Серверная часть:** хостинг и серверы, бэкенд журнала пропусков, веб-интерфейс УК, мобильные интеграции. Бюджет инфраструктуры — в NDA-материалах.
- **Партнёрские технологии:** партнёрство с 10 крупнейшими производителями домофонов; программа «Госуслуги.ДОМ Ready».
- **Поддержка/эксплуатация:** Service Desk, LMS, инструменты управления монтажными бригадами и приёмочных тестов.
- **Юридический контур:** оформление Оператора ЕСД, соглашение с Оператором ГИС ЖКХ/Госуслуги.Дом, типовые договоры с УК, ОСС-документация, согласование с Минцифры.

## Нефункциональные требования и ограничения

- **Безопасность и ИБ:** криптозащищённый протокол; аудит ИБ; SLA на доступность сервиса; резервные каналы открытия (на случай сбоев BLE/сети).
- **Защита персональных данных:** соответствие 152-ФЗ; минимизация хранимых данных о жителях; согласия через ЕСИА.
- **Регуляторика:** соответствие нормативке ЖКХ, требованиям антитеррора, программам «Умный город» и цифровизации ЖКХ.
- **Масштабируемость:** архитектура должна выдерживать федеральный рост — от 100–300 объектов на старте до 5 000+ за 24 месяца, с горизонтом до 3 млн пользователей и 10+ регионов.
- **Открытость:** открытое API для производителей домофонов (Госуслуги.ДОМ Ready), white-label для крупных девелоперов.
- **Экономические ограничения (структурно):** OPEX фиксирован и не растёт пропорционально масштабу; CAC стремится к нулю за счёт распространения через УК и дилерскую сеть. Конкретные цифры — в NDA-материалах.
- **Организационные ограничения:** подключение УК требует ОСС → необходима автоматизация шаблонов и методическая поддержка.

### Этапы

- **Этап 1. MVP (0–3 мес, июнь–сентябрь 2026).** Запуск электронного журнала пропусков; интеграция кнопки «Заказать пропуск» в Госуслуги.Дом; типовые документы (договор УК, шаблон ОСС); подготовка интеграций с API Госуслуги.Дом.
- **Этап 2. Пилоты (3–6 мес, сентябрь–декабрь 2026).** Подключение 100–300 домов и 100+ УК; ≥ 20 000 заказов пропусков; среднее время реакции УК ≤ 15 минут; ≥ 3 партнёра-производителя ПАК; пилот распознавания авто-номеров.
- **Этап 3. Масштабирование (6–12 мес, декабрь 2026 – июнь 2027).** 500+ объектов, 300 000+ пользователей, ≥ 70 % заявок без диспетчера; LMS для монтажников и УК; маркетплейс сертифицированных монтажников; партнёрство с 10 крупнейшими производителями домофонов; интеграции с курьерами/такси/охраной. Финансовые KPI — в NDA-материалах.
- **Этап 4. Монетизация (12–24 мес, июнь 2027 – июнь 2028).** Платные подписки УК, продажа ПАК, биржа установок, B2B-интеграции с Yandex/OZON/WildBerries и сервисными службами; изменение нормативной базы и обязательность использования УК.
- **Этап 5. Национальное масштабирование (18–24 мес, декабрь 2027 – июнь 2028).** ≥ 5 000 объектов, ≥ 3 млн пользователей, ≥ 10 регионов с активными проектами, ≥ 3 крупных партнёрства с муниципалитетами/застройщиками; white-label для девелоперов; экосистема устройств, аналитика, видеонаблюдение, голосовой доступ. Финансовые KPI этапа (выручка, EBITDA, оценка компании) — в NDA-материалах.

### Параллельный регуляторный трек

Параллельно с продуктовым развитием идёт **трек изменения нормативной базы и обязательности использования платформы УК**: согласование с Минцифры/Минстроем, продвижение нормативных актов, по которым подключение УК к единой платформе становится обязательным или экономически обоснованным. Этот трек ведёт GR/Public Affairs-функция — детали в `regulatory-roadmap.md`. Без него масштабирование на этапах 4–5 упирается в добровольный onboarding и низкую конверсию ОСС.

### Шесть направлений масштабирования (трек B)

После выхода на этап 5 платформа масштабируется по шести измерениям одновременно:

- **по регионам** (расширение из пилотных регионов к федеральному покрытию);
- **по застройщикам** (партнёрства с девелоперами для предзакладывания совместимых ПАК в новостройки);
- **по видам домов** (МКД → коттеджные посёлки → бизнес-центры);
- **по поставщикам заявок** (расширение круга B2B-партнёров);
- **по функциональности** (голосовой доступ, видеонаблюдение, аналитика, white-label);
- **по технологическим партнёрам** (расширение реестра сертифицированных производителей ПАК и монтажников).

### Ближайшие шаги (организация работ)

1. Создание рабочей группы; определение технических, организационных и юридических сторон взаимодействия; формирование соглашения о совместной работе Оператора ИС и Оператора ЕСД. (Финансовая часть и Term Sheet — отдельный трек, материалы в NDA.)
2. Подготовка ТЗ на доработки **ГИС ЖКХ** и **Госуслуги.Дом** (функциональность, структуры данных, API, интерфейсы).

## Высоконагруженная архитектура (ESD ServiceDesk)

> Этот раздел описывает целевую архитектуру **программного контура** Оператора ЕСД — серверной платформы, через которую обрабатываются пропуски, события доступа, ОСС-голосования, биллинг УК, B2B-интеграции с поставщиками заявок и устройства ПАК. Подробное функциональное ТЗ — `functional-spec.md`. Модель данных — `data-model.md`. Карта API — `api-map.md`. Нагрузочные характеристики — `load-profile.md`.

### Целевые масштабы (горизонт 24 мес)

- 5 000+ объектов в проде, 3+ млн пользователей, 10+ регионов РФ.
- Ожидаемые нагрузки: ~12 М проходов/сутки, ~2 М пропусков/сутки, пик ACL-проверок до 1 400 RPS, пик B2B-заявок до 3 000 RPS.
- 1+ млн событий-проходов в сутки требуют outbox-pattern + Kafka и ClickHouse для OLAP.

### Каналы интерфейсов

| Канал | Аудитория | Технология |
|---|---|---|
| **BFF Mobile** для Госуслуги.Дом | Жители (через embed-кнопку «Заказать пропуск» в приложении ПАО МРТС) | REST/JSON; интеграция через партнёрское API ЕПГУ (СМЭВ-4) |
| **BFF Web Operator** | УК-администраторы, диспетчеры, охрана | Vue 3.5 + PrimeVue 4 + Pinia 3 + Vite 6 (REUSE из pass24-servicedesk) |
| **Public B2B Gateway** | Поставщики заявок (Yandex GO/Маркет, OZON, WildBerries, Купер, Самокат, Delivery, Dostavista) | REST + Webhooks + mTLS + JWT (Client Credentials) |
| **Device Gateway** | ПАК-устройства (BLE-домофоны, контроллеры) | mTLS + MQTT + custom binary protocol; Go-сервис для low-latency |
| **OEM Gateway «Госуслуги.ДОМ Ready»** | Производители домофонов (BAS-IP, SLINEX, Beward, Tantos и др.) | Open REST API с публичной OpenAPI-спецификацией |
| **Installer BFF** | Монтажники биржи + LMS | REST/JSON, мобильное приложение монтажника |

### Сводная матрица REUSE / UPGRADE / NEW (vs pass24-servicedesk)

| Узел | pass24-servicedesk | ESD ServiceDesk | Обоснование |
|---|---|---|---|
| Background tasks | `lifespan`-хуки (Telegram, IMAP polling, SLA-watcher, Bitrix24 sync, CSAT) | Kafka topics + Temporal.io workflows + dedicated workers (arq/Celery) | На 1 М+ событий/сутки `lifespan` теряет события при рестарте, нет ретраев и DLQ |
| База данных | PostgreSQL single-node | PostgreSQL Citus (шардинг by `object_id`) + read replicas | 5 000+ объектов × 100+ пропусков/мес = 500k+ строк/мес только в `passes` |
| Кэш | отсутствует | Redis Cluster (3 master × 2 replica) | ACL-проверки 5k req/sec target требуют кэша |
| Аналитика | Postgres-агрегаты в `stats/` | ClickHouse (отдельная витрина) + Metabase/Superset | Журнал проходов = append-only timeseries 1М+/сутки |
| Auth | bcrypt + JWT HS256 | ЕСИА (OAuth2/OIDC + GOST 2012) + внутренний JWT RS256 | Госконтур требует ЕСИА (см. ADR-002) |
| Async-протоколы | REST + WebSocket `/ws` | REST + Webhooks + Kafka events + MQTT (для устройств) | Устройства и B2B-партнёры требуют push-механизмов |
| Deployment | docker-compose + 1 VPS | Kubernetes multi-AZ + ArgoCD GitOps | Аккредитация ИС ГИС ЖКХ требует доказанной отказоустойчивости |
| Crypto | bcrypt | КриптоПро CSP 5.0 + HSM 2.0 + XAdES-T (ADR-005) | 152-ФЗ + ФСТЭК-17 |
