# Операционные плейбуки ESD ServiceDesk

## Введение

Документ содержит 5 операционных плейбуков — сценарные регламенты для команд продаж, сопровождения, инженерного отдела, Service Desk, B2B-партнёрств и Customer Success. Каждый плейбук описывает шаги, ответственных, SLA, артефакты и KPI.

Плейбуки используются:
- При обучении новых сотрудников (см. LMS-курсы в `architecture.md` → модуль knowledge-svc).
- В Service Desk как референс при обработке инцидентов.
- В аудитах процессов (квартальный review).

Связанные документы:
- `architecture.md` — компоненты, на которые опираются плейбуки (Журнал пропусков, ПАК, Партнёрский шлюз).
- `functional-spec.md` — детальное ТЗ модулей.
- `regulatory-roadmap.md` — юридическая часть онбординга УК (договор, ОСС).
- `NDA/team-plan.md` (вне репозитория) — роли (CSM, Head Sales, Head Eng, L1/L2, B2B Partnerships) и RACI.
- `NDA/tariff-model.md` (вне репозитория) — pricing-логика для шага «КП» в плейбуках.

## Плейбук 1. Подключение УК

**Триггер:** входящий лид (ОП outbound, реферал, заявка с сайта) или активный outbound.

| Шаг | Кто | SLA | Артефакт |
|---|---|---|---|
| 1. Квалификация (BANT + кол-во объектов/квартир) | KAM | ≤ 1 раб. день | Запись в Bitrix24 |
| 2. Демо-встреча (15 мин видео + 30 мин live) | KAM + Solution Engineer | ≤ 5 раб. дней | Demo-deck, video-rec |
| 3. Pricing предложение (тариф по матрице из `NDA/tariff-model.md`) | KAM | ≤ 2 раб. дня после демо | Commercial proposal PDF |
| 4. Юридическое согласование договора | Юрист УК ↔ наш юрист | ≤ 10 раб. дней | Подписанный договор |
| 5. Подготовка ОСС (если требуется по 44-ст. ЖК РФ) | onboarding-менеджер + конструктор ОСС | 30–45 дней (по закону) | Протокол ОСС |
| 6. Оформление мастер-аккаунта УК + ЕСИА УК-админа | onboarding | 1 день | Активный аккаунт |
| 7. Загрузка реестра объектов / помещений / жителей (CSV или ГИС ЖКХ pull) | onboarding + DataOps | ≤ 5 раб. дней | Заполненный кабинет |
| 8. Обучение УК (LMS-курс «УК-админ» 4 ч + сертификация) | L&D | 1 неделя | Сертификат |
| 9. Soft-launch журнала пропусков (выкл. ПАК, только ручные пропуска) | onboarding + L1 | 1 неделя | Метрики первых 100 пропусков |
| 10. Hard-launch (ПАК on, биллинг on) | onboarding + Eng | После приёмки шага 9 | Подписанный акт |
| 11. Hand-off в CSM | onboarding → CSM | 14 дней наблюдения | Health-score карточка |

**KPI плейбука:**
- Time-to-Live ≤ **60 дней** (без ОСС), ≤ **90 дней** (с ОСС).
- NPS первого контакта ≥ **40**.

**Особенности:**
- Если УК уже на действующей лицензии (191-ФЗ) и подключение не требует решения собственников (например, по типовому контракту управления), шаг 5 пропускается — TTL сокращается на 30–45 дней.
- При работе с холдингом УК (≥ 10 объектов) шаги 6–11 параллелизуются по группам объектов; CSM обязательно dedicated.

## Плейбук 2. Установка ПАК

**Триггер:** заявка от УК-админа из кабинета на установку ПАК.

| Шаг | Кто | SLA | Артефакт |
|---|---|---|---|
| 1. Заявка на установку (из кабинета УК) | УК-админ | — | Тикет «installation» |
| 2. Обследование объекта (offline visit или photo-survey через моб. приложение монтажника) | Монтажник из биржи | ≤ 3 раб. дня | Чек-лист обследования |
| 3. КП на оборудование + работы | Eng-back office | ≤ 2 раб. дня | КП |
| 4. Закупка ПАК + логистика | Закупки + лог. партнёр | ≤ 10 раб. дней | Накладная |
| 5. Назначение монтажника (биржа автоматически по гео + рейтингу) | LMS-engine + Head Eng | ≤ 1 день | Назначен исполнитель |
| 6. Обучение монтажника по модели ПАК (мини-курс LMS, обяз. ≥ 80 % теста) | LMS | ≤ 1 день | Допуск к работе |
| 7. Монтаж + ПНР | Монтажник | 4–8 ч | Фото-отчёт + видео-приёмка |
| 8. Приёмочные тесты (10 точек: ЕСИА → BLE → открытие → лог) | L2 + Монтажник | 1 ч | Acceptance protocol |
| 9. Hand-off в Service Desk (объект «активен») | L2 | — | Запись в реестре ПАК |
| 10. Гарантия и ТО (12 мес default) | Service Desk | — | TO-карточка |

**KPI плейбука:**
- Среднее время от заявки до hard-launch ПАК ≤ **14 раб. дней**.
- First-time fix rate ≥ **90 %**.
- Отказ в первый месяц ≤ **2 %**.

**Особенности:**
- Photo-survey через моб. приложение монтажника (шаг 2) — приоритетный режим для большинства объектов; offline-визит требуется только при технологически сложных установках (мульти-дверь, многоточечный шлагбаум, интеграция с действующей СКУД через WebPass-Gate).
- Acceptance-protocol (шаг 8) подписывается ЭЦП (или сканом с подписью УК-админа); хранится в audit-svc как XAdES-T.
- Биржа монтажа автоматически назначает монтажника по гео-удалённости (≤ 50 км) + рейтингу (≥ 4,0) + сертификации (Pro/Expert для мульти-точек).

## Плейбук 3. Обработка инцидента «не открылся проход»

**Триггер:** push от жителя/охранника / алерт мониторинга / звонок в Service Desk.

```
T+0    -> Жилец нажимает "сообщить о проблеме" в моб. приложении
T+30s  -> AI-ассистент пытается self-heal:
           - re-issue BLE-токен;
           - проверка сети ПАК;
           - проверка статуса жителя/пропуска
        Если решено -> кейс закрыт, лог.
T+1m   -> Если нет решения -> эскалация L1 (chat + push оператору)
T+2m   -> L1 проверяет: статус ПАК (online/offline), последний успешный лог,
          конфликт пропусков. Может удалённо открыть точку доступа (для
          подтверждённого жителя) с регистрацией в ИБ-журнале.
T+5m   -> SLA L1: первый ответ. Если не закрыто -> L2.
T+15m  -> L2 (инженер) подключается:
           - SSH в ПАК (если поддерживается);
           - анализ BLE-логов;
           - переконфигурация.
T+1h   -> Если не решено -> выезд полевого инженера через биржу монтажа,
          приоритет CRITICAL, ETA <= 4 ч в городе.
T+24h  -> Postmortem обязателен для CRITICAL. Запись в БЗ.
```

### Фоллбэки

- **ПАК offline > 5 мин** → автоматический режим «открыто по списку белых» (заранее загруженные мастер-ключи + охранник вручную сверяет).
- **Полная потеря связи** → охранник работает по бумажному журналу пропусков, выгрузка на печать каждые 4 ч.

### KPI плейбука

- L1 first response ≤ **5 мин** (24/7).
- MTTR CRITICAL ≤ **4 ч**.
- P1 incidents/мес/объект ≤ **0,5**.

### Особенности

- Self-heal на T+30s закрывает по статистике PASS24 ~40–55 % инцидентов без эскалации (re-issue BLE-токена — наиболее частая операция).
- Удалённое открытие в шаге T+2m фиксируется в audit-svc как событие «manual override by L1» с обоснованием и идентификатором подтверждённого жителя — это юр. значимое действие.
- Postmortem (T+24h) — обязательная запись в knowledge-svc; раз в квартал — review всех CRITICAL postmortems со стороны CTO/CISO.

## Плейбук 4. Подключение партнёра-поставщика заявок (B2B)

**Триггер:** outbound CRO к крупному поставщику заявок (Yandex GO/Маркет, OZON, WildBerries, Купер, Самокат, Delivery, Dostavista) или входящая заявка от партнёра.

| Шаг | Кто | SLA |
|---|---|---|
| 1. NDA + краткий BANT (Y/OZ/WB-уровень) | CRO | 1 нед |
| 2. Security review партнёра (опросник 152-ФЗ + ФСТЭК + arch-review) | CISO + Юрист | 2 нед |
| 3. Договор + DPA (data processing agreement) | Юрист × 2 | 3–4 нед |
| 4. Технический онбординг: sandbox API, выпуск API-ключей трёх уровней | DevRel + Lead BE | 1 нед |
| 5. UAT на 1 пилотном объекте УК (white-list) | CSM + партнёр | 2 нед |
| 6. Прод-релиз с rate-limit ramp-up (10 → 100 → 1000 req/min за 14 дней) | SRE | 2 нед |
| 7. Comm-launch + co-marketing | Маркетинг + партнёр | — |

**KPI плейбука:**
- Time-to-live партнёра ≤ **90 дней**.
- Доля партнёрской выручки в общем revenue — целевое значение в `NDA/financial-model.md`.

**Особенности:**
- Security review (шаг 2) включает проверку партнёра как контрагента ПДн: где хранит данные курьеров, как передаёт данные в ESD, есть ли согласие курьеров на участие в системе пропусков ESD.
- DPA (шаг 3) — кастомизированный для каждого партнёра; для Уровня 3 (Full Enterprise) — индивидуальный SLA с штрафами за нарушение.
- Rate-limit ramp-up (шаг 6) — критичен для предсказуемой нагрузки на partners-svc; первые 14 дней DevRel мониторит метрики p99 latency и error rate, ограничивая партнёра при отклонениях.

## Плейбук 5. Поток заявки от партнёра до открытия точки доступа

**Триггер:** курьер партнёра подъезжает к МКД и вызывает API запроса доступа.

```
[Партнёр]                                        [ЕСД API GW]                   [ПАК]
  | POST /v1/access-request {object, slot, courier_id, oneTimeCode} |
  |  (HMAC-signed, mTLS)                                            |
  |---------------------------------------------------->            |
  |                                                                  |
  |             [Auth+RateLimit+ABAC]                                |
  |                v                                                 |
  |             [Биллинг: проверка квоты партнёра]                   |
  |                v                                                 |
  |             [Согласование с УК-policy:                           |
  |               - временное окно;                                  |
  |               - макс. кол-во курьеров/день;                      |
  |               - разрешённые подъезды]                            |
  |                v                                                 |
  |             [Если требуется явное согласие жителя ->             |
  |              push в моб.-кабинет, ожидание 60 сек,               |
  |              fallback "deny by default"]                         |
  |                v                                                 |
  |             [Issue BLE one-time token, TTL 30 мин]               |
  | <----- 200 OK + token ---                                        |
  |                                                                  |
  |               [Курьер подходит к точке]                          |
  |                                  -- BLE handshake -------------> |
  |                                                                  |
  |                                  [ПАК offline? ->                |
  |                                   локальная проверка             |
  |                                   подписи токена офлайн]         |
  |                                                                  |
  |                                  [Открытие + запись в журнал]    |
```

### Фоллбэки

- **ПАК offline** → локальная подпись токена (Ed25519) с TTL 24 ч и кэш белого списка (последняя синхронизация ≤ 5 мин назад).
- **Партнёрский API недоступен** → партнёр заранее получает batch-токены на 24 ч с rate-cap (например, 50 ключей/час). Используется в случае деградации `partners-svc` или сети до партнёра.
- **Курьер потерял связь** → охранник в моб. приложении видит «ожидаемый курьер» (карточка с фото + код), открывает вручную, лог пишется как «manual confirmation».

### KPI плейбука

- ≥ **99,5 %** успешных открытий (success rate).
- < **0,01 %** инцидентов мошенничества (отказ от подтверждения, использование чужого токена, попытка прохода после revoke).

### Особенности

- HMAC-подпись + mTLS — обязательный уровень защиты партнёрского API; ECDSA-подписи токенов на стороне ПАК с ECC P-256 / GOST 2012.
- Согласие жителя — конфигурируется по политике УК; для большинства жилых домов default — «явное согласие при первом курьере + автосогласие на повторных».
- TTL one-time token = 30 мин по default; может быть переопределён партнёром при выпуске (минимум 5 мин, максимум 2 ч).
- Запись в журнал прохода (`AccessEvent` в ClickHouse) — XAdES-T-подписана; webhook `pass.used` отправляется партнёру в течение 1 сек после фактического прохода.
