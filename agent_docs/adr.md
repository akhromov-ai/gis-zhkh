## [2026-04-29 14:30] ADR-001: Запуск ЕСД как отдельного Оператора с интеграцией в Госуслуги.Дом и ГИС ЖКХ

### Статус
Принято

### Контекст
Цифровизация контроля доступа в жилые дома пересекается одновременно с тремя крупными контурами:

- государственный (ГИС ЖКХ как источник адресных справочников, данных об УК и жителях);
- пользовательский (мобильное приложение «Госуслуги.Дом» как массовая точка входа жителя);
- индустриальный (производители домофонов, монтажные бригады, поставщики заявок Yandex/OZON/WildBerries и т. п.).

Без выделенного оператора каждая УК и каждый производитель решает задачу доступа собственными несовместимыми решениями. Это блокирует появление сквозных пропусков для курьеров/служб и не даёт государству инструмента ограничения доступа в жилой фонд (антитеррор).

### Решение
Создать отдельную информационную систему — **Оператор «Единой Среды Доступа» (ЕСД)** — и развивать её как федеральную инфраструктурную платформу:

1. Точкой входа для жителя становится кнопка **«Заказать пропуск»** внутри **Госуслуги.Дом** (без отдельного приложения).
2. Серверная часть Оператора ЕСД ведёт **электронный журнал пропусков** и предоставляет двусторонние API в ГИС ЖКХ и в Госуслуги.Дом.
3. На стороне дома — собственный **ПАК** (BLE, криптозащищённый протокол, облачная СКУД) и реестр сертифицированных монтажников; для УК без СКУД — облачное решение «под ключ».
4. Подключение УК идёт через **ОСС** с максимально автоматизированными шаблонами решений.
5. Сервисная модель комбинированная: подписочный канал для УК + разовая выдача ПАК + биржа монтажа + B2B-интеграции с поставщиками заявок.

### Обоснование

- **Государственная синергия:** интеграция с Госуслуги.Дом и ГИС ЖКХ снимает проблему доверия пользователей и даёт распространение через государственный канал.
- **Платформенный эффект:** единый стандарт пропусков делает возможной интеграцию с курьерскими и сервисными службами федерального уровня (Yandex, OZON, WildBerries и др.) — ни одна отдельная УК этого не достигнет.
- **Регуляторика:** отдельный Оператор удобнее интегрируется с Минцифры и нормативной базой (152-ФЗ, ЖКХ, антитеррор), чем «совмещённый» сервис внутри Госуслуги.Дом.
- **Открытость:** открытое API и программа «Госуслуги.ДОМ Ready» позволяют сразу подключить 10 крупнейших производителей домофонов, не выстраивая собственного аппаратного производства с нуля.

### Последствия

**Положительные:**

- Чёткие границы ответственности между Оператором ГИС ЖКХ, Оператором Госуслуги.Дом и Оператором ЕСД.
- Масштабируемость до 5 000+ объектов и 10+ регионов без переписывания продукта.
- Появление управляемого канала B2B-интеграций с партнёрами-поставщиками заявок.

**Отрицательные / риски:**

- Зависимость от сроков и приоритетов Оператора Госуслуги.Дом и Оператора ГИС ЖКХ по доработке API и UI.
- Подключение УК требует ОСС — длительный и неуправляемый напрямую процесс.
- Риск сопротивления УК и недоверия жителей; необходима системная методическая поддержка.
- Регуляторные риски (ПДн, согласие Минцифры).
- Зависимость от партнёров-производителей ПАК на старте (≥ 3 к пилоту, 10 к 12 месяцу).

### Альтернативы рассмотрены

- **Сервис внутри Госуслуги.Дом без отдельного оператора.** Отклонено: невозможно масштабировать аппаратную часть, биржу монтажа и B2B-интеграции в рамках госприложения; конфликт с регуляторикой.
- **Локальные решения УК / коробочные СКУД.** Отклонено: не дают единого стандарта пропусков и не интегрируются с курьерскими службами федерального уровня.
- **Только программная платформа без ПАК.** Отклонено: продажа ПАК — ключевой компонент монетизации и обеспечения единого технического стандарта на устройствах.

## [2026-04-29 15:00] ADR-002: Технологический стек ESD ServiceDesk

### Статус
Принято

### Контекст
Платформа должна выдержать пиковые нагрузки 5 000 RPS на API Gateway, до 1 400 RPS на ACL-проверки (BLE-критично, p99 < 100 ms) и 3 000 RPS на B2B-выдачу ключей курьерам в часы доставки. Параллельно нужно ускорить time-to-market для MVP (3 мес) и максимально переиспользовать готовый production-ready код.

### Решение
- **Backend, CRUD-сервисы:** Python 3.12 + FastAPI 0.115 + SQLModel + asyncpg + Alembic + Pydantic 2 (REUSE стека pass24-servicedesk).
- **Backend, горячие пути (`acl-svc`, `device-gateway`, `partners-svc`):** Go 1.22 — ради низкой latency и эффективного управления горутинами при 5–10k RPS на ноду.
- **Frontend:** TypeScript + Vue 3.5 + Vue Router 4 + Pinia 3 + PrimeVue 4 (Aura) + Vite 6 + ECharts 6 (REUSE дизайн-системы и сборки pass24-servicedesk).
- **Mobile:** React Native или Flutter — выбор отложен в Фазе 1 до получения ТЗ от ПАО МРТС по embed-кнопке (если SDK от Оператора Госуслуги.Дом — выбираем под него).
- **Async pipeline:** Kafka (3 broker × 2 AZ), NATS JetStream (для BLE-событий), Redis Cluster (cache + locks + Streams), Temporal.io (long-running ОСС/онбординг).
- **Data layer:** PostgreSQL 16 + Citus (шардинг by `object_id`), ClickHouse (3 шарда × 2 реплики) для журнала проходов, TimescaleDB для метрик устройств, Qdrant cluster для RAG, S3 RuStack (VK Cloud / Yandex / Selectel) для медиа.
- **AI:** Claude Sonnet 4 (Anthropic SDK) — основной LLM; YandexGPT / GigaChat — fallback для регуляторных сценариев (ОСС, юр. вопросы) на случай блокировок Claude.
- **Infrastructure:** Docker + Kubernetes (Yandex MK8S или VK Cloud K8s), ArgoCD GitOps, GitHub Actions CI/CD (REUSE workflow из pass24-servicedesk).
- **Service Mesh:** Istio (mTLS внутри кластера, distributed tracing, требование ФСТЭК-17).
- **API Gateway:** Kong (или KrakenD) + PT Application Firewall как WAF.
- **Crypto:** КриптоПро CSP 5.0 R3 + КриптоПро HSM 2.0 + КриптоПро TSA + плагин CryptoPro EDS Browser Plug-in (для УК-кабинета).
- **Observability:** Prometheus / VictoriaMetrics + Grafana + Loki + Tempo + OpenTelemetry + Sentry + MaxPatrol SIEM.

### Обоснование
- **REUSE pass24-servicedesk даёт MVP за 3 мес.** В pass24 уже работают 8 модулей (auth, tickets, knowledge, customers, projects, notifications, assistant, stats, telegram), 40+ endpoints, 238 тестов, готовая дизайн-система Vue3/PrimeVue. Форк ускоряет старт на 4–6 месяцев.
- **Гибрид Python + Go.** Python даёт скорость разработки CRUD-сервисов, Go — необходимую производительность на горячих путях (ACL-проверка p99 < 100 ms нереализуема на Python+FastAPI при 5 000 RPS).
- **Citus + ClickHouse — реакция на отличие от pass24.** В pass24 single-node PostgreSQL держит ~26 пользователей, ~100 тикетов. Наш масштаб на 4 порядка больше — без шардинга и OLAP-витрины не выдержит.
- **Kafka + Temporal вместо `lifespan`-хуков.** В pass24 фоновые задачи (Telegram dispatcher, IMAP polling, SLA-watcher, Bitrix24 sync, CSAT) запущены в `lifespan` приложения. Это не масштабируется и теряет события при рестарте. На федеральном масштабе обязательны Kafka topics с DLQ и Temporal workflows для ОСС-голосования (10 дней) и онбординга УК (мульти-этапного).
- **K8s multi-AZ обязателен** для аккредитации ИС в ГИС ЖКХ как «Commercial IS» — ФСТЭК-17 требует доказанной отказоустойчивости.

### Последствия
**Положительные:**
- Time-to-market MVP сокращён на 4–6 мес за счёт REUSE pass24-servicedesk.
- Архитектура выдерживает целевой масштаб 5000+ объектов и 12 М проходов/сутки.
- Соответствие требованиям ФСТЭК-17 и 152-ФЗ заложено архитектурно.
- Open-source стек минимизирует vendor lock-in (за исключением КриптоПро — это требование регулятора).

**Отрицательные:**
- Гибрид Python+Go повышает требования к найму (нужны senior'ы обоих стеков).
- Kubernetes multi-AZ повышает OPEX инфраструктуры на 30–40 % vs single-node.
- КриптоПро лицензии и HSM — единоразовый CAPEX (точная сумма — в NDA-материалах, статья «КриптоПро HSM 2.0 + лицензии»).

### Альтернативы рассмотрены
- **Только Python (без Go).** Отклонено: ACL p99 < 100 ms нереализуем при 5k RPS на FastAPI без Go-побочки.
- **Полный rewrite с нуля на Go или Rust.** Отклонено: time-to-market 9–12 мес вместо 3.
- **Managed-СУБД вместо Citus.** Рассмотрено (Yandex Managed PostgreSQL): подходит для read replicas, но шардинг по `object_id` нужно делать самостоятельно.
- **Eventually consistent NoSQL (Cassandra/MongoDB).** Отклонено для основной БД: требования к транзакционной целостности (биллинг, ОСС, audit).

## [2026-04-29 15:05] ADR-003: Тарифная модель ESD ServiceDesk (наследие PASS24 ADR-001)

### Статус
Принято

### Контекст
Нужна предсказуемая, прозрачная и масштабируемая модель монетизации SaaS-подписки УК, продажи ПАК и B2B-API партнёрам. PASS24.online в своей собственной БЗ (репозиторий `pass24-knowlege-base`, ADR-001 v1.1) уже отработал рабочую модель «жёсткие лимиты с прозрачным апгрейдом» — это контрастирует с непрозрачным «pay-as-you-go» (как у мобильных операторов) и с overpriced enterprise-only моделями.

### Решение (структурно)
Перенять модель PASS24 ADR-001 v1.1 с адаптацией под доменику ЕСД. Структурно фиксируется следующее (конкретные цены, проценты комиссии, лимиты и доплаты — в NDA-материалах, документ `NDA/tariff-model.md`; разработчик читает их из конфигурации `billing-svc`, а не из этого ADR):

- **4 уровня подписки УК:** Базовый «УК Старт», Стандартный «УК Видео», Профессиональный «УК Полный», Мультиобъектный «Холдинг».
- **Доплаты ко всем тарифам:** SMS, голосовой ассистент, white-label, дополнительные точки доступа, расширенный SLA, ОСС-конструктор расширенный.
- **ПАК (разовая выручка):** три модели (Lite / Standard / Pro), плюс установка через биржу и платное ТО после гарантийного периода.
- **B2B-API партнёрам — 3 уровня:** Read / RW + Webhooks / Full Enterprise, с разными правами и моделями оплаты.
- **Биржа монтажа:** комиссия ESD с заказа монтажа (стандарт и instant payout — разные ставки), комиссия с ТО, бесплатный LMS-курс для допуска и платная сертификация Pro/Expert.
- **Логика апгрейда (REUSE PASS24 ADR-001):** ≥ 80 % лимита → push УК-админу + e-mail + звонок CSM; 100 % → soft-block на 24 ч + предложение апгрейда / overage; overage-ставки определены в NDA.
- **Целевая структура revenue:** заложена с приоритетом подписок УК и продажи ПАК как ядра монетизации.

Реализация — модуль `billing-svc` (NEW). Все числовые параметры (тарифы, лимиты, комиссии, overage-ставки) хранятся в защищённом конфиге, источник правды — `NDA/tariff-model.md`.

### Обоснование
- **REUSE проверенной модели PASS24** снижает риск ошибки в ценообразовании на старте.
- **Жёсткие лимиты с прозрачным апгрейдом** дают единый «источник истины» для биллинга, минимизируют спорные ситуации с УК.
- **SMS как доплата** (а не часть тарифа) корректно отражает переменные затраты на SMS-шлюз.
- **3 уровня B2B-API** позволяют масштабировать revenue от партнёров без переписывания контрактов: Yandex GO начинает с Read, перерастает в RW, при больших объёмах переходит во Full.
- **Маржа на ПАК** заложена с учётом локализации производства в РФ через сертифицированных партнёров (BAS-IP / Sigur / Bolid).

### Последствия
**Положительные:**
- Прозрачная цена для УК → ускоряет цикл сделки.
- Простая биллинг-логика → меньше код-сложности в `billing-svc`.
- Линейная масштабируемость revenue.
- Конкретные тарифы хранятся отдельно от технической документации — легче править ценообразование без релиза кода.

**Отрицательные:**
- Клиенты, привыкшие к «безлимит» от PASS24, требуют индивидуального обсуждения при миграции — нужен переходный период.
- Жёсткие квоты должны балансировать между оттоком (если слишком тесные) и стагнацией (если слишком широкие) — требуется регулярный пересмотр на основе данных (ежеквартально).

### Альтернативы рассмотрены
- **Pure usage-based (за каждый пропуск).** Отклонено: непрозрачно для УК, сложно для биллинга, нет компаундинга.
- **Enterprise-only (договорные цены).** Отклонено: сжимает воронку — мелкие УК (4–5 объектов) отсеиваются.
- **Freemium.** Отклонено: УК — B2B-сегмент, freemium тут не работает (УК принимают решение через ОСС, freemium-флоу не помогает).

## [2026-04-29 15:10] ADR-004: Жёсткая стратегия Госуслуги.Дом-only канала распространения

### Статус
Принято (по решению пользователя 2026-04-29 в Plan-режиме)

### Контекст
Для жителей платформа доступна как **«кнопка «Заказать пропуск»» в мобильном приложении Госуслуги.Дом**. Это закрытое API ПАО МРТС / Минцифры. Альтернатива — параллельно разрабатывать собственное PWA / мобильное приложение под брендом ESD, чтобы снизить регуляторную зависимость.

Пользователь зафиксировал: **жёсткая стратегия — только Госуслуги.Дом-канал, без параллельного own-app**.

### Решение
- Все B2C-сценарии для жителей реализуются исключительно через embed-интеграцию в Госуслуги.Дом по партнёрскому API ЕПГУ (СМЭВ-4, регламент 1.4 от 09.11.2022).
- Собственное PWA/нативное приложение под брендом ESD **не разрабатывается** в горизонте 24 мес (опция остаётся как стратегический fallback при срыве партнёрки, но без бюджета и без roadmap).
- Партнёрский трек переговоров с **ПАО МРТС / Минцифры** становится критическим путём проекта № 1.
- Веб-кабинет УК / Диспетчера / Охраны — собственный, не зависит от Госуслуги.Дом (это B2B-канал).
- Веб-витрина для жителей (общая информация о сервисе, маркетинг) — собственная, без функций заказа пропусков.

### Обоснование
- **Единый канал** даёт стопроцентную интеграцию в существующий пользовательский флоу — жителю не нужно скачивать второе приложение.
- **Доверие пользователей** к госприложению выше, чем к частному стартапу.
- **CAC ≈ 0** через государственный канал распространения (нет затрат на привлечение пользователей жилья — они уже там).
- **Согласие на ОПД** проще оформить через ЕСИА-аутентификацию.
- **Ускорение MVP-цикла** на 2–3 мес за счёт отсутствия параллельной разработки моб. приложения.
- **Концентрация бюджета** на core-сервисах и регуляторике вместо распыления на mobile-разработку.

### Последствия
**Положительные:**
- Единый источник user growth — органическая база Госуслуги.Дом (миллионы жителей уже зарегистрированы).
- Меньше mobile-команды (–2 FTE iOS/Android в Фазе 2).
- Меньше расходов на App Store / Google Play / RuStore (нет отдельного приложения).

**Отрицательные / критические:**
- **Регуляторный риск № 1:** срыв или затягивание партнёрского соглашения с ПАО МРТС → блок основного канала distribution → весь go-to-market буксует.
- **Юридические ограничения:** Госуслуги.Дом — государственный продукт, сложно вносить продуктовые изменения по нашему усмотрению.
- **Брендирование:** ESD-бренд не виден жителю напрямую — для public awareness нужен отдельный контентный канал (сайт, СМИ, B2B-presence).
- **Зависимость:** при изменении политики Минцифры по партнёрскому API → потенциальная потеря канала.

### Компенсирующие меры
1. **GR-функция в штат не позднее мес. 0.** Один FTE GR/Public Affairs с прямым выходом на ПАО МРТС / Минцифры / ДИТ Москвы — критическая роль из RACI.
2. **Эксклюзивные предсоглашения с 1–2 производителями домофонов** в Фазе 0 как «moat» против конкурентов (A-Access, Simplgate, VisitorControl).
3. **Pre-аудит ФСТЭК** за 2 мес до офиц. подачи на аккредитацию ИС.
4. **Резервная стратегия (без бюджета):** в случае срыва партнёрки в течение 6 мес — экстренная разработка собственного PWA для жителей (лаг 3–4 мес, дополнительный бюджет — параметр в NDA-материалах). Решение принимает Совет директоров.

### Альтернативы рассмотрены
- **Гибридная стратегия (Госуслуги.Дом + own-app fallback).** Отклонено пользователем: распыление бюджета и команды; снижение фокуса на партнёрке.
- **Сначала own-app, потом Госуслуги.Дом.** Отклонено пользователем: дольше путь к госмасштабу, конкуренты (PASS24/A-Access) могут раньше получить partnership.

## [2026-04-29 15:15] ADR-005: Immutable audit log с XAdES-T-подписью каждой записи

### Статус
Принято

### Контекст
Журнал проходов (`access_events`), голоса ОСС (`oss_vote`) и ключевые операции (выдача/отзыв ключей, изменения биллинга) должны быть **юридически значимыми** и иметь свойство **неотказуемости** (non-repudiation). Это требование:
- ФСТЭК приказ № 17 (для ГИС класса К2) — мера ОЦЛ (контроль целостности).
- 152-ФЗ — необходимость доказать факт согласия / действия пользователя.
- Жилищный кодекс РФ — протокол ОСС должен иметь признаки юр. документа для регистрации в ГИС ЖКХ.
- Антитеррор — журнал проходов может потребоваться органам.

### Решение
- **Каждая запись `AccessEvent`** в ClickHouse подписывается XAdES-T (XML Advanced Electronic Signature with Timestamp) с использованием GOST 2012 (Stribog + ECDSA-256).
- **Каждый голос `OSSVote`** подписывается ЕСИА-сертификатом голосующего (через плагин CryptoPro EDS в браузере или через мобильный embed).
- **Итоговый протокол ОСС** (PDF) подписывается ЭЦП Оператора ЕСД (XAdES-T).
- **Audit log ключевых операций** (выдача/отзыв ключа, изменение тарифа, доступ оператора к ПДн) — отдельный сервис `audit-svc` с immutable append-only хранилищем + XAdES-T-подпись каждой записи.
- **Источник подписей:** КриптоПро HSM 2.0 (физический модуль) для корневых ключей оператора ЕСД; КриптоПро TSA для timestamp.
- **Производительность:** XAdES-T verify CPU-bound, при пике ОСС-голосования (~500 RPS) выносится в отдельный пул воркеров с горизонтальным масштабированием. Очередь Kafka topic `audit.sign`.

### Обоснование
- **Юридическая значимость** — единственный способ обеспечить, чтобы протокол ОСС, подписанный через ЕСИА, был принят как полноценный юр. документ.
- **Неотказуемость** — пользователь не может оспорить «я не голосовал»: подпись содержит timestamp + сертификат ЕСИА.
- **Целостность журнала проходов** — критично при инцидентах (например, проникновение); запись в журнале защищена от подделки.
- **Соответствие ФСТЭК-17** — без XAdES-T аккредитация ИС как «Commercial IS» невозможна.
- **REUSE pass24-knowlege-base** — нет, NEW целиком (в pass24 audit log хранится в Postgres без подписей).

### Последствия
**Положительные:**
- Юр. значимость протоколов ОСС → возможность регистрации в ГИС ЖКХ как полноценных решений собственников.
- Соответствие ФСТЭК-17 К2 → аккредитация «Commercial IS» возможна.
- Стандарт качества для регуляторов и инвесторов (можно показать «у нас всё подписано GOST 2012»).

**Отрицательные:**
- Единоразовый CAPEX на КриптоПро HSM 2.0 и постоянный OPEX лицензий КриптоПро CSP/TSA. Конкретные суммы — в NDA-материалах (`investment-plan.md`).
- Усложнение производительности: XAdES-T верификация на пике 500 RPS требует ~10–20 ядер CPU только под подпись. Требуется отдельный пул воркеров.
- Усложнение DevOps: HSM требует физического хостинга или сертифицированного облачного HSM-as-a-Service (доступно в Yandex Cloud, VK Cloud).

### Альтернативы рассмотрены
- **Hash-chain без XAdES-T.** Отклонено: не имеет юр. силы по российскому законодательству.
- **Внешний TSA (свободный, не GOST).** Отклонено: ФСТЭК-17 требует ГОСТ-сертифицированного TSA.
- **Подпись только итогового протокола ОСС, без подписи каждого голоса.** Отклонено: при оспаривании результата невозможно доказать конкретный голос.

## [2026-04-29 15:20] ADR-006: RBAC и набор ролей системы

### Статус
Принято

### Контекст
В pass24-servicedesk используются 4 роли (RESIDENT, PROPERTY_MANAGER, SUPPORT_AGENT, ADMIN). Этого достаточно для service-desk-сценария, но не для ESD: у нас появляются устройства (ПАК), партнёры B2B (Yandex/OZON), монтажники (биржа), охранники, диспетчеры, ИБ-сотрудники, регуляторный комплаенс. Нужно расширенное RBAC под все каналы (mobile / web operator / B2B / device / installer) и под все организационные функции.

### Решение
Принять следующий набор из **16 ролей**:

**Внешние (клиентские):**
1. **RESIDENT** — житель (через Госуслуги.Дом).
2. **PROPERTY_MANAGER** — УК-администратор (веб-кабинет).
3. **DISPATCHER** — диспетчер УК (веб, обработка пропусков).
4. **SECURITY_GUARD** — охранник на проходной (веб + терминал).
5. **INSTALLER** — монтажник биржи (моб. приложение + LMS).
6. **PARTNER_API** — B2B-партнёр (M2M-клиент, Yandex GO/OZON/WB и др.).
7. **GUEST** — гость с одноразовым пропуском (без аккаунта, через QR/deep link).
8. **DEVICE** — ПАК-устройство (M2M, mTLS-клиент с привязкой к серийному номеру).

**Внутренние (сотрудники Оператора ЕСД):**
9. **L1_SUPPORT** — оператор поддержки L1 (Service Desk, 24/7).
10. **L2_SUPPORT** — инженер поддержки L2.
11. **SALES_MANAGER** — менеджер ОП (продажи УК).
12. **CS_MANAGER** — менеджер ОС (сопровождение УК).
13. **MARKETING** — маркетолог (доступ к контентным разделам, без ПДн).
14. **FINANCE_BILLING** — финансовый/биллинг-сотрудник.
15. **SECURITY_OFFICER** — ИБ-сотрудник, CISO, аудитор.
16. **IT_ADMIN** — ИТ-администратор инфраструктуры.

**Регуляторный (read-only внешний):**
17. **MINTSIFRY_AUDITOR** — представитель Минцифры / Регулятора (read-only выгрузки агрегированной статистики, без ПДн).

> Технически 17, но в `functional-matrix.md` `MINTSIFRY_AUDITOR` рассматривается как разновидность external-роли с минимальным доступом и в основной матрице сводится в столбец «Регуляторный комплаенс» вместе с внутренней compliance-ролью.

**Привязка ролей к каналам:**
- BFF Mobile (Госуслуги.Дом): RESIDENT, GUEST.
- BFF Web Operator: PROPERTY_MANAGER, DISPATCHER, SECURITY_GUARD, L1_SUPPORT, L2_SUPPORT, SALES_MANAGER, CS_MANAGER, MARKETING, FINANCE_BILLING, SECURITY_OFFICER, IT_ADMIN, MINTSIFRY_AUDITOR.
- Public B2B Gateway: PARTNER_API.
- Device Gateway: DEVICE.
- Installer BFF: INSTALLER.

**Иерархия наследования:**
- ADMIN из pass24 → разделяется на IT_ADMIN (инфраструктура) и SECURITY_OFFICER (ИБ).
- SUPPORT_AGENT из pass24 → разделяется на L1_SUPPORT и L2_SUPPORT.

Подробная матрица доступа CRUDX по 14 модулям — `functional-matrix.md`.

### Обоснование
- **DDD bounded contexts** требуют явного разделения ролей (житель ≠ УК ≠ охранник ≠ партнёр).
- **152-ФЗ + ФСТЭК-17** требуют принципа least privilege: каждый сотрудник имеет доступ только к данным своего bounded context.
- **B2B-партнёры и устройства** — отдельные классы M2M-клиентов; их нельзя смешивать с пользовательскими ролями.
- **Регулятор** должен иметь возможность получать агрегированные данные без доступа к ПДн жителей.

### Последствия
**Положительные:**
- Чёткая модель доступа → проще проходить ФСТЭК-аудит.
- Возможность разделить операции по командам (ОС не видит финансы, ОП не видит ИБ-журналы).
- Гибкость наращивания ролей (можно добавить REGIONAL_DIRECTOR, ENTERPRISE_CSM в Фазе 4 без переписывания всего).

**Отрицательные:**
- Сложность frontend-логики (router guards, conditional rendering).
- Необходимость поддерживать 16 ролей в тестах E2E.
- При найме новых функций (например, WHITE_LABEL_ADMIN) — добавление новой роли требует ADR-update.

### Альтернативы рассмотрены
- **Оставить 4 роли pass24 + flag-поля.** Отклонено: не масштабируется, нарушает least privilege.
- **ABAC (Attribute-Based Access Control) вместо RBAC.** Рассмотрено: гибче, но избыточно для MVP. Решение — стартовать с RBAC, по мере роста добавлять ABAC-атрибуты (например, region, object_id) как фильтры поверх ролей.
- **Микро-роли (Pass.Read, Pass.Create, Pass.Revoke и т. д.).** Отклонено: фрагментирует модель, увеличивает сложность ревью прав.
