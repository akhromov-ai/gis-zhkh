# Регуляторная дорожная карта ESD ServiceDesk

## Введение

Документ описывает технические регуляторные требования и последовательность процедур, необходимых для запуска и эксплуатации платформы ESD ServiceDesk в государственном контуре РФ. Согласно [ADR-004](adr.md), канал распространения для жителей — встраивание в Госуслуги.Дом, поэтому регуляторный трек (аккредитация ИС в ГИС ЖКХ, аттестация по ФСТЭК-17, подключение к СМЭВ-4) находится на критическом пути проекта.

Документ описывает:
- последовательность регуляторных процедур по месяцам с тех. ответственными;
- параллельные не-блокирующие технические треки;
- регуляторные требования по каждому контуру (ЕСИА, ФСТЭК, 152-ФЗ, КриптоПро, аккредитация ИС, партнёрский API);
- технический критический риск из ADR-004 и компенсирующие меры на стороне разработки.

Связанные документы: [adr.md](adr.md) (ADR-004), [risks-register.md](risks-register.md), [architecture.md](architecture.md).

## Раздел 2. Параллельные не-блокирующие треки

Технические процедуры, не находящиеся на критическом пути, но обязательные для соответствия регуляторике:

| Трек | Месяц | Тех. ответственный | Описание |
|---|---|---|---|
| Уведомление РКН о начале обработки персональных данных | 0 | Регуляторный комплаенс | Подача уведомления как Оператор ПДн (152-ФЗ ст. 22). Срок рассмотрения 30 дней. Обязательно до первого пилотного запуска. |

| Проверка лицензий УК-партнёров (191-ФЗ) | непрерывно с фазы 2 | onboarding tooling team | Самой ESD лицензия УК НЕ требуется (платформа не управляет МКД), но при подключении каждой новой УК автоматически проверяется наличие действующей лицензии в реестре Государственной жилищной инспекции через ГИС ЖКХ или ОГВ-открытые данные. |

## Раздел 3. Регуляторные требования

### 3.1. ЕСИА (Единая система идентификации и аутентификации)

| Параметр | Требование |
|---|---|
| Регистрация ИС | Обязательная регистрация ESD как ИС в личном кабинете на https://esia.gosuslugi.ru/ через Минцифры. Получение `client_id` / `client_secret`. |
| Криптоалгоритмы | GOST Р 34.11-2012 (Stribog, хеш) + GOST Р 34.10-2012 (ECDSA-256, подпись). Поддержка GOST Р 34.10-2001 (legacy) — только для миграции старых систем до конца 2026. |
| Формат подписи | PKCS#7 detached в UTF-8. Подписываемые параметры: `scope`, `timestamp`, `clientId`, `state` без разделителей. Кодировка результата — Base64 URL-safe. |
| Сертификат | Должен быть выдан аккредитованным УЦ (КриптоПро или эквивалент). Должен быть предварительно зарегистрирован в ЕСИА и привязан к ИС. **Самоподписанные сертификаты не поддерживаются с 01.04.2020.** |
| Протокол | OAuth 2.0 / OpenID Connect 1.0 (текущий стандарт с 2022). SAML 2.0 — deprecated, новые системы не подключаются. |
| Уровни УЗ пользователя | Простая (логин/пароль), Стандартная (с верификацией паспорта), Подтверждённая (с очной верификацией). Для ESD требуется **Подтверждённая УЗ** для действий, требующих юр. значимости (ОСС-голосование, выдача постоянных ключей). |
| Webhook-уведомления о смене статуса учётной записи | Подписка через `userinfo` endpoint, кэш на 15 мин в Redis. |

### 3.2. ФСТЭК-17 (для ГИС) и ФСТЭК-21 (для ИСПД)

| Параметр | Требование |
|---|---|
| Класс защищённости | **1Г** (ИСПДн с большим объёмом данных более 100 000 субъектов). Требуется аттестация по ФСТЭК-17 как ГИС. |
| Меры защиты (ФСТЭК-17) | ИАФ.1–ИАФ.7 (идентификация и аутентификация), УПД.1–УПД.13 (разграничение доступа), АВЗ (антивирус), СОВ (система обнаружения вторжений), ОЦЛ (контроль целостности), ЗИС (защита информационной системы), ЗСВ (защита среды виртуализации — для K8s-кластеров). |
| Меры защиты (ФСТЭК-21, ИСПД) | ПДн обрабатываются с УЗ-2 (если есть биометрия — фото/Face ID) или УЗ-3 (если только ФИО, телефон, email). Для ESD по умолчанию УЗ-2 (есть Facial Recognition в дорожной карте). |
| Программно-аппаратные средства | **PT Application Firewall** (WAF), **PT Application Inspector** (статический анализ кода), **MaxPatrol SIEM** (мониторинг событий ИБ), **PT Sandbox** (анализ файлов). Антивирус с сертификатом ФСТЭК (Kaspersky Endpoint Security или Dr.Web Enterprise Security Suite). |
| СОВ | Сертифицированная СОВ (например, Континент-IDS или PT NAD). |
| Аудит | Лицензированный лицензиат ФСТЭК проводит аудит и выдаёт аттестат соответствия. Срок аттестата — 5 лет, ежегодный контрольный аудит. |
| Сегментация | Сеть разделена на сегменты: DMZ (внешние API), внутренний (микросервисы), управление (admin-доступ), хранение данных. mTLS между сегментами через Service Mesh (Istio). |

### 3.3. 152-ФЗ + Постановление Правительства № 1119 + ФСБ № 378

| Параметр | Требование |
|---|---|
| Уведомление в РКН | Подача уведомления о начале обработки ПДн как Оператор ПДн. Срок рассмотрения 30 дней. |
| Согласие субъекта ПДн | Получается через документы, оформляющие правоотношения УК ↔ собственник (ЖК РФ ст. 162) или Устав ТСЖ/кооператива. Отдельное письменное согласие 152-ФЗ не требуется (ст. 6 ч. 1 п. 5). |
| Уровень защищённости ПДн | **УЗ-2** (есть биометрия — facial recognition / голосовая идентификация) или **УЗ-3** (только классические ПДн). Определяется в Модели угроз. |
| Локализация | Все ПДн граждан РФ должны храниться на серверах, физически расположенных на территории РФ. Облачные провайдеры — только с сертификатом 152-ФЗ (VK Cloud, Yandex Cloud, Selectel, MailRu Cloud). |
| Регуляторный орган | Роскомнадзор — надзор за соблюдением 152-ФЗ. ФСТЭК — за технические меры защиты. ФСБ — за криптосредства. |
| Минимизация данных | Хранить только то, что необходимо. Для ESD: ФИО, телефон, email, номер квартиры, ЕСИА-ID. Не хранить: СНИЛС, паспортные данные, данные о доходах. |
| Шифрование | At-rest — AES-256 (для не-госконтура) или GOST 28147-89 / Кузнечик (для ФСТЭК). In-transit — TLS 1.3. ПДн в логах — маскируются (last4 phone, hash email). |

### 3.4. КриптоПро SDK / GOST 2012 / XAdES-T

Технические компоненты криптографического стека, используемые в backend-сервисах и на устройствах ПАК.

| Компонент | Описание |
|---|---|
| КриптоПро CSP 5.0 R3 | Криптопровайдер для backend-сервисов (`identity-svc`, `audit-svc`, `oss-svc`). Сертифицирован ФСБ. Лицензируется на каждое серверное ядро. |
| КриптоПро HSM 2.0 | Физический модуль безопасности для хранения корневых ключей платформы. Используется для подписи прошивок ПАК и итоговых протоколов ОСС. Минимум 2 устройства в HA-паре в разных AZ. |
| КриптоПро TSA | Сервис меток времени (Time-Stamp Authority) для XAdES-T-подписей. Обеспечивает юридическую неотказуемость каждого подписанного события. |
| CryptoPro EDS Browser Plug-in | Браузерный плагин для подписи ЭЦП в кабинете УК (протоколы ОСС). Поддерживает Chrome, Firefox, Edge. |
| Библиотеки backend | `pycryptopro` (Python wrapper), `cprocsp-pycades` (Python для CADES/XADES), `KalkanCrypt-bridge` (Java для XAdES-T). |
| Формат подписи аудит-логов | XAdES-T (XML Advanced Electronic Signature with Time-stamp). Каждое `AccessEvent` и `OSSVote` подписываются и снабжаются меткой времени КриптоПро TSA. |
| Ротация ключей | Ключи XAdES-T-подписи ротируются раз в 12 мес. Корневые ключи в HSM — раз в 5 лет. |

## Раздел 4. Технический критический риск из ADR-004 и компенсирующие меры

ADR-004 фиксирует технологическую зависимость проекта от партнёрского API ЕПГУ / Госуслуги.Дом — без параллельного own-app для жителей. Это означает: задержка получения партнёрского доступа напрямую блокирует выход MVP к жителям.

### 4.1. Технические компенсирующие меры

| Мера | Срок внедрения | Тех. ответственный |
|---|---|---|
| Pre-аудит ФСТЭК за 2 мес до офиц. подачи | мес. 1 .. 3 | CISO + лицензиат ФСТЭК |
| Полный пакет тех. документов для аккредитации готов на мес. 2 | мес. 0 .. 2 | CTO + Регуляторный комплаенс |
| Параллельная разработка Web Operator Cabinet и Service Desk монтажа | непрерывно с фазы 1 | Platform team + Domain team |
| Архитектура без жёсткой привязки к закрытому API: сервисы `passes-svc`, `acl-svc`, `device-gateway` работают независимо от наличия канала Госуслуги.Дом | непрерывно | Core team + IoT team |
| Тестовый контур через «Own IS» до получения «Commercial IS» | мес. 1 .. 7 | Regулaторный комплаенс + CTO |

### 4.2. Технический trip-wire по ADR-004

Если на мес. 6 заявка на партнёрский API ЕПГУ не подана или находится без движения более 90 дней, требуется внеплановый технический review с участием CTO + CISO + Регуляторного комплаенса для оценки альтернатив и фиксации нового технического ADR (ADR-007). Возможные исходы:
1. Продление ожидания на 3 мес с актуализацией пакета документов.
2. Запрос пересмотра ADR-004 (расширение скоупа разработки на own-front-end fallback для жителей).
3. Pivot на B2B2C-модель: API для крупных девелоперов с их собственными приложениями (white-label).

Решение фиксируется как новый ADR.
