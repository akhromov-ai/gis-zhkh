# Концепция и связи Оператора ЕСД

> Верхнеуровневое описание идеи, участников и связей сервиса «Единая Среда Доступа». Документ читается **первым** при входе в проект — он отвечает на вопросы «что это», «кто здесь действует» и «как они связаны». Технические детали — в `architecture.md`, `functional-spec.md`, `data-model.md`. Финансовая модель и тарифы — в `NDA/` (вне репозитория).

## 1. Суть продукта

**«Единая Среда Доступа» (ЕСД)** — национальная платформа цифрового контроля доступа в жилые дома. Технически это:

- **Электронный журнал пропусков** — серверная часть Оператора ЕСД, в которой регистрируются все заявки на доступ, статусы и история проходов.
- **Кнопка «Заказать пропуск»** — встроенный раздел в мобильном приложении «Госуслуги.Дом»; единственная точка входа для жителя.
- **Программно-аппаратный комплекс (ПАК)** на BLE-домофонах и контроллерах — реализует физическое открытие точек доступа по криптозащищённому протоколу.
- **Открытое API** для производителей домофонов («Госуслуги.ДОМ Ready»), партнёров-поставщиков заявок (курьерские службы и маркетплейсы) и управляющих компаний.
- **Биржа сертифицированных монтажников + LMS** — сеть установщиков и обучение, контроль качества монтажа.

Платформа объединяет государственный контур (ЕСИА, ГИС ЖКХ, Госуслуги.Дом), индустриальный (производители ПАК, монтажники) и партнёрский (Yandex, OZON, WildBerries и другие поставщики заявок) под единым стандартом цифрового пропуска.

### 2.2. Описание сторон

| Сторона | Что делает | Чем взаимодействует с ЕСД |
|---|---|---|
| **Житель** | Заказывает пропуск, голосует на ОСС, открывает дверь по приближению | Кнопка «Заказать пропуск» в Госуслуги.Дом, ЕСИА-аутентификация |
| **Управляющая компания (с собственной СКУД)** | Получает заявки на доступ через приложение/портал и обрабатывает их в своей системе | Заявка из Госуслуги.Дом → УК; через мобильное приложение/портал УК |
| **Управляющая компания (без СКУД)** | Использует облачную СКУД ЕСД для постоянного и временного доступа | Портал заявок ЕСД + облачная СКУД для жильцов и сотрудников |
| **Оператор ЕСД** | Ведёт журнал, выдаёт цифровые ключи, держит интеграции, управляет биржей монтажа и Service Desk | Серверная часть, сертифицированные API, ПАК-устройства |
| **ГИС ЖКХ** | Источник адресных справочников, данных об УК, жителях, местах доступа | Двусторонний API (доработка SOAP) + расширение справочников «места доступа» (шлагбаумы, подъезды, калитки) |
| **Госуслуги.Дом** | Канал доступа жителя; UI пропусков; шаблоны документов; ОСС | Доработка UI и функций; партнёрское API ЕПГУ |
| **Партнёры-поставщики заявок** | Запрашивают «сквозные» одноразовые ключи для своих курьеров (Yandex GO/Маркет/Такси/Доставка, OZON, WildBerries, Купер, Самокат, Delivery, Dostavista, сервисные и ремонтные службы) | Public B2B Gateway (REST + Webhooks + mTLS) |
| **Производители домофонов** | Поставляют сертифицированные устройства, совместимые с ЕСД | OEM Gateway «Госуслуги.ДОМ Ready», открытая OpenAPI-спецификация |
| **Сертифицированные монтажники** | Устанавливают ПАК на объектах УК | Биржа монтажа + LMS + Service Desk |
| **Охрана** | Проверяет личность гостя/курьера на проходной | Web-терминал + аутентификация через ЕСИА |
| **Государство (Минцифры/Минстрой)** | Регулятор; задача в рамках программ антитеррора и «Умный город» | ЕСИА, ФСТЭК-17, ИС ГИС ЖКХ как «Commercial IS», СМЭВ-4 |
| **Сервисные организации** | Получают единый интерфейс для доступа к разным домам разных УК | Портал заявок + B2B-API |

### 2.3. Что меняется в окружении

- **Госуслуги.Дом:** доработка UI, функций и шаблонов документов; добавление раздела «Заказать пропуск» и страниц ОСС.
- **ГИС ЖКХ:** доработка API; расширение справочников «места доступа» (шлагбаумы, подъезды, калитки) и связанных потоков (заявки, данные о доступе).
- **Серверная часть Оператора ЕСД:** новый модуль с журналом пропусков, API-шлюзом, ОСС-конструктором, биллингом, биржей монтажа и аудит-журналом.

## 3. Базовые сценарии

### 3.1. Сценарий «Заказ пропуска жителем»

1. Житель в Госуслуги.Дом нажимает «Заказать пропуск» → формирует заявку.
2. Заявка попадает в **журнал пропусков ЕСД**; жителю/гостю возвращается идентификатор/QR/BLE-ключ.
3. Параллельно ЕСД отправляет **e-mail в УК** с информацией о необходимости обеспечить доступ.
4. УК фиксирует обработку в журнале (или работает по своей СКУД); информация о проходе возвращается жителю.
5. Накопление неисполненных пропусков и e-mail-запросов **создаёт давление на УК** (см. § 3.4 «Стратегия входа»).

### 3.2. Сценарий «Курьер / такси / доставка»

1. Партнёр (Yandex GO, OZON, WildBerries и другие) при оформлении доставки вызывает **API ЕСД** на выпуск разового ключа.
2. ЕСД формирует **«сквозной» временный ключ** на нужный набор точек доступа (шлагбаум + калитка + подъезд) и автоматически отправляет его в приложение курьера при выдаче заказа.
3. На месте — открытие через BLE-приближение или прямое соединение «смартфон ↔ контроллер» по криптозащищённому протоколу.
4. При необходимости — **проверка личности охраной** через аутентификацию ЕСИА.
5. Журнал прохода фиксируется в ЕСД; webhook о проходе отправляется партнёру.

Это **минимизирует возможности для конфликтов и хищений** (заранее известный курьер, фиксированный временной слот, идентификация через ЕСИА).

### 3.3. Сценарий «УК без собственной СКУД»

1. УК подключается к **облачной СКУД ЕСД** через ОСС.
2. Через портал УК или мобильное приложение жильцам и сотрудникам выдаются **постоянные ключи** с привязкой к мастер-ключу.
3. УК выполняет **групповое администрирование** (массовая выдача/отзыв/обновление ключей) — например, при заселении/выселении.

### 3.4. Стратегия входа на рынок (важная особенность)

Сервис может стартовать **до подключения УК** — жители уже могут заказывать пропуски через Госуслуги.Дом, ЕСД будет автоматически отправлять e-mail УК. Это создаёт следующую динамику:

1. **Накопление просроченных пропусков и e-mail с запросами** на адрес УК → УК вынужденно начинает работать с журналом.
2. **Недовольство жителей неисполненными пропусками** → жители инициируют **ОСС по подключению Журнала пропусков УК** (а потом и ПАК).
3. Чтобы запустить эту воронку быстро, **ОСС-процесс должен быть максимально автоматизирован** — шаблоны решений, методички вовлечения жителей, цифровое голосование через Госуслуги.Дом с ЕСИА-подписью.

Технически это означает, что модули `passes-svc`, `notifications-svc` и `oss-svc` должны быть готовы **раньше** полного onboarding-флоу `uk-svc` (см. матрицу этапов в `functional-matrix.md`).

## 4. Технология доступа: «нулевое время доступа»

Концепт пользовательского опыта — открытие двери **без действий пользователя**. Реализуется тремя элементами:

1. **Автоматическое открытие по приближению** — для смартфонов и пассивных меток.
2. **Прямое BLE-соединение** «смартфон ↔ контроллер» (без обязательного интернета на стороне устройства в момент прохода).
3. **Криптозащищённый запатентованный протокол** обмена данными.

Технические детали (Ed25519/ECDSA, оффлайн-валидация, локальный кэш ключей в EEPROM устройства) — в `functional-spec.md`, модули `acl-svc` и `device-gateway`.

## 5. Дорожная карта (концептуально)

> **Календарная привязка:** мес. 0 = июнь 2026 (старт MVP); полная шкала — в `architecture.md`, раздел Roadmap → «Календарная привязка». Все «мес. N» в документах проекта отсчитываются от этого ориентира.

### 5.1. Развитие сервиса (трек A)

1. **Старт сервиса** — запуск электронного журнала пропусков.
2. **Устройства и технологические партнёры** — партнёрство с 10 крупнейшими производителями домофонов.
3. **LMS и курсы по устройствам + Service Desk** — обучение и поддержка работы монтажников.
4. **Управление бригадами и приёмочные тесты при монтаже** — управление монтажом и контроль качества.
5. **Партнёры — поставщики заявок** — интеграция с курьерскими и сервисными службами (Yandex GO/Маркет, OZON, WildBerries, Купер, Самокат, Delivery, Dostavista, сервисные и ремонтные службы).

Подробная разбивка по версиям и этапам — в `architecture.md` (раздел Roadmap) и `functional-matrix.md` (матрица «Модули × Этапы»).

### 5.2. Регулирование (трек A1, параллельный)

- Изменение нормативной базы и обязательность использования УК подключения к платформе. Этот трек ведёт GR/Public Affairs (см. `regulatory-roadmap.md`).

### 5.3. Масштабирование (трек B)

Сервис «Единая Среда Доступа» масштабируется по шести измерениям:

- **по регионам** — постепенное расширение из пилотных регионов к федеральному покрытию;
- **по застройщикам** — белые партнёрства с девелоперами для закладывания совместимых ПАК на этапе сдачи новостроек;
- **по видам домов** — многоквартирные, коттеджные посёлки, бизнес-центры;
- **по поставщикам заявок** — расширение круга B2B-партнёров;
- **по функциональности** — голосовой доступ, видеонаблюдение, аналитика, white-label;
- **по технологическим партнёрам** — расширение реестра сертифицированных производителей ПАК и монтажников.

## 6. Ценность сервиса для каждой стороны

| Сторона | Ценность |
|---|---|
| **Управляющая компания** | Перенос ответственности на жителя (автор пропуска), простой и контролируемый доступ курьеров, повышение собираемости задолженности, групповое администрирование и обновление ключей |
| **Жители** | Удобство оформления доступа, временные и постоянные пропуска с привязкой к мастер-ключу |
| **Производители домофонов** | Открытая архитектура для интеграции, новый рынок устройств — программа «Госуслуги.ДОМ Ready» |
| **Государство** | Ограничение и упорядочивание доступа в жилые дома в рамках программы антитеррора и цифровизации ЖКХ |
| **Сервисные организации** | Единый интерфейс для организации процесса доступа к разным домам разных УК — без необходимости договариваться с каждой УК отдельно |
| **Партнёры-поставщики заявок** | Сквозной цифровой ключ, минимизация конфликтов и хищений, интеграция через единый API вместо точечных договорённостей |

(Финансовая часть — отдельный трек, материалы в `NDA/`.)
2. **Техническое задание на доработки ГИС ЖКХ и Госуслуги.Дом.** Описание необходимых доработок функциональности, структур данных, API, интерфейсов для реализации нового сервиса.
